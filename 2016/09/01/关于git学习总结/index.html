#关于git学习总结

###1. git和svn最大的区别 ：
这两者最大的区别在于svn的结构是集中式管理结构，而git的结构是分布是管理结构。

其次是git多了一个本地版本库，我们在写好代码之后通过commit命令是提交至本地的版本库，然后再通过PUSH命令将本地的版本库提交至共享版本库。

而对于svn来说，我们在写好代码的时候用add和commit命令是直接将代码提交至共享本版库（服务器），没有本地版本库这一说法。

就因为没有本地版本库，所以svn在没有网络的情况下是无法将代码提交至服务器的，而git可以先将自己写好的代码提交至本地的版本库中。

###2. git的优点：

1. 在很多情况下，git的速度比svn的速度要快很多。
2. svn使用分支是比较笨拙的，但是git可以轻松的拥有无限个分支。
3. svn必须联网才能工作，而git可以在没有网络的情况下将代码提交至本地版本库中。

###3. git基本指令解析

  1. 初始化一个代码仓库

    • git init
      这条指令会在自定义目录中创建一个本地代码版本仓库。

  2. 给GIT配置用户名和邮箱

    • git config user.name “XMG”

    • git config user.email “XMG@163.com”
      这两条语句分别配置用户名和邮箱，git必须先进行配置，才能进行commit，否则会出现错误，这个用户名和邮箱只是对当前的本地仓库进行了配置，并没有对所有目录下的本地仓库进行配置，这意味着以后如果在别的目录中创建了一个仓库，那么这个仓库必须再次进行用户名和邮箱的配置.
      这个配置信息可以在本地版本库中(.git)中的config中进行查看。

    • git config –global user.name “XMG”

    • git config –global user.email “XMG@163.com”
      这是配置全局的用户和邮箱。那么不管你在哪里创建你本地版本库，里面都会有你的用户和邮箱信息，这个比上面的更有效。我喜欢。

  3. 初始化项目(这里以创建一个main.m为例)

    • touch main.m
    • git add main.m
      将main.m添加到git的暂缓区
    • git commit -m “注释”
      这里就将暂缓去的所有内容都提交到了本地版本库中，然后清空了暂缓区。
      这里需要注意，我们添加或者是修改了main.m（或者是其他的文件）都需要先将这些文件添加到暂缓区，然后才能提交到本地版本库中。

  4. 查看文件状态指令

    • git status
      这里如果文件名为红色，那么表示该文件被添加或者被修改，但还没有被添加到暂缓区。
      如果为绿色，那么表示该文件在暂缓区，但是还没有提交到本地版本库。

  5. 给git指令起别名

    • git config alias.st “status”
      这条指令表示对status起一个st别名，这个不是全局的版本库配置，只对当前的本地版本库有效。

    • git config alias.ci “commit -m”
      对commit -m起一个别名ci，这也是一个当前本地版本库的配置

    • git config –global alias.st “status”
      对status起一个全局的别名st，这个对所有的本地版本库都有效。

  6. 删除文件
    • git rm main.m
      这条指令是将main.m从本地工作目录中删除了，并且这个时候在暂缓区也有对应的删除状态了，我们可以通过status指令进行查看，发现这里的deleted: main.m为绿色，我们就可以进行提交了，以改变本地的版本库内容。
  7. 查看版本信息
    • git log
      这里的版本号是由40位的哈希值组成
    • git reflog
      一般我们版本回退会使用这个指令，这个指令更加简洁和高效的显示出了版本信息。

  8. 版本回退

    • git reset –hard HEAD
      表示回到当前的版本，如果你在当前的版本中进行了修改，还没有提交到本地版本库，那么使用这条指令就可以恢复到还没有修改时候的文件。
    • git reset –hard HEAD^
      表示回退到上一个版本
    • git reset –hard HEAD^^
      表示回退到上上一个版本
    • git reset –hard HEAD~100
      表示回退到前100个版本
    • git reset –hard HEAD 版本号
      表示回到对应的版本号，这里的版本号一般取前5位，如果出现了前5位相同的情况，那么就去前6位。依次类推。
    • git push -f
      强制性的将本地版本库中的代码提交到共享版本库中

      如果进行版本的回退，一般我们选定好一个版本以后先使自己的本地版本库回退到想要的版本(使用git reset….指令),然后我们需要使用git push -f 来强制的使共享版本库中的版本回退到和自己本地版本库相同的版本，然后共享版本库的事情就做好了，其他开发人员想要pull共享版本库的时候必须也得将自己的本地版本库中的代码会退到想要的版本，然后再pull。否则会出现pull下来之后，新版本的代码还在的情况。如果先回退到旧版本，然后pull就不会这样了。

  9. 版本备份

    • 项目经理初始化项目，发布任务，要求张三开始项目weibo1.0的开发，张三经过奋战，完成了weibo1.0版本的开发，然后提交当前的版本到共享版本库中，项目经理从共享版本库中更新代码，查看代码，满意之后将weobo1.0进行tag备份，并上传到AppStore，以下是备份的操作指令：

      git tag -a weibo1.0 -m “这是weibo1.0版本的备份”
      这里weibo1.0是备份版本的名字，”*”中间的内容注释。这里需要注意，现在的版本备份只是在本地版本库中有了weibo1.0的备份，在共享版本库中还没有这个备份，所以我们应该使用指令：
      git push origin weibo1.0
      将备份版本提交到共享版本库中，供张三及其他开发人员pull。

    • 项目经理发布weibo2.0版本的开发，让开发人员继续2.0版本的开发，这个时候1.0版本突然出现了严重的bug，必须马上解决，项目经理上阵，首先在本地的某一目录中创建一个文件夹，用来存放备份的1.0的代码。然后进行git clone，现在的代码和最新版的一样，我们得使用指令git checkout weibo1.0让代码回到1.0原始的样子。
      这时候不能马上进行代码的修复，因为如果修改了备份的代码，那么我们后期就不能通过tag标签切换到原始好的1.0版本的代码了，所以我们应该再创建一个分支，这个分支中会再拷贝出备份的代码，这样在分支中修改代码就不会影响到tag标记的原备份代码了。创建并切换到分支的指令如下:

      git checkout -b weibo1.1fixBug

    • 项目2.0的开发持续进行中，当经理修复好的1.0的bug时，就对当前的分支进行commit并且push到分支中的本地版本库，然后bug修复完成，并且上传到AppStore，然后再次备份到分支本地的版本库中。这次备份的名字不能和以前的备份名字，还有分支的名字相同。使用

      git tag -a weibo1.1 -m “修复了1.0bug的1.1版本”
      然后使用git push origin weibo1.1将这个版本备份同步到共享的版本库中。

    • 开始和2.0项目合并,我们在原来主分支中的经理代码中使用pull，将pull目标分支选择为修复bug的分支，然后pull下来之后就合并了原代码和修复bug后的代码了，如果这里有冲突的话就直接解决就行。合并后提交并且push到共享额版本库中，以供其他的开发人员pull。

    • 合并完成，开始删除分支。

      使用git branch或者git branch -r可以查看当前的分支。
      这里应该显示的是fixBug那个分支(绿色的就是当前分支)然后切换到master分支，使用git checkout master切换。开始删除指令:

      git branch -d weibo1.1fixBug
      git branch -r -d origin/weibo1.1fixBug
      git push origin –delete weibo1.1fixBug
      必须使用这三条语句，删除三个地方的分支，这样下次pull的时候就不会再本地的版本库中再次出现weibo1.1fixBug分支了。
